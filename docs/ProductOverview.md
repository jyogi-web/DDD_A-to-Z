# ゲーム企画書 — Lang War

## 世界観

プログラミング言語や技術スタックが、ギルドとして存在する世界。ユーザーは自分の推し技術を選び、現実のGitHub活動によってそのギルドを支援する。

コードを書くことが、そのままゲーム内の貢献になる。CommitやPull RequestはContributionとして解析され、Contribution Point（CP）に変換される。CPが積み上がることで、ギルドの勢力ランキングや領土が変化していく。

このゲームの核は、**現実のGitHub活動がゲームを動かすこと**。

## コア体験

ユーザーが体験する一番大事な流れは以下。

```text
開始
↓
推し技術（ギルド）を選ぶ
↓
GitHubでコードを書く
↓
ContributionがCPに変換される
↓
勢力ランキング・領土が変化する
↓
推しギルドを勝たせる
↓
シーズン終了時に最強言語が決定する
```

プレイヤーは、単に自分の活動量を眺めるのではなく、「自分の開発が推しギルドの勝敗に影響している」と感じられることを目指す。

## MVPのコア体験

MVPでは、最終形の「言語戦争シミュレーション」全体を作らない。まず検証する体験は次の1つに絞る。

> GitHubで書いたコードが、自分の推しギルドのCPになり、勢力ランキングが動くのを見る。

最初のプロダクトは、複雑な戦略ゲームではなく、推し技術を支援する総選挙型のスコアボードとして成立させる。ユーザーがGitHub連携を行い、直近の活動がCPに変換され、ギルドのランキングと自分の貢献ログに反映されるところまでをMVPとする。

MVPでできること。

- GitHubログイン
- GitHubのCommit / Pull Request取得
- 使用言語集計
- Contribution解析
- CP変換
- 推しギルドへのCP加算
- 勢力ランキング表示
- 簡易的な領土・勢力図の表示

MVPで扱うポイントは、最初はCommitとPull Requestに限定する。Issue close、Star、能力値補正、ギルド施設、キャラクタースキルは、コア体験が成立してから追加する。

## 用語

| 用語 | 意味 |
|---|---|
| ギルド | 特定のプログラミング言語・技術を支持するプレイヤーコミュニティ |
| 推し技術 | ユーザーが支援するギルド。MVPでは1ユーザー1つだけ選ぶ |
| Contribution | GitHub上で行われる開発活動。Commit、Pull Requestなど |
| CP | Contribution Point。Contributionをゲーム内ポイントに変換したもの |
| 勢力値 | ギルドに集まったCPの合計 |
| 領土 | 地理的な土地ではなく、その技術分野における主導権・覇権の可視化 |
| シーズン | 一定期間で区切られた競争単位 |

## ゲームの核心ルール

GitHubの活動がContributionとして解析され、CPに変換される。CPはユーザーが所属する推しギルドに加算され、ギルドごとの勢力値になる。

ポイントの重みづけ案は以下。

| アクション | CP |
|---|---:|
| Commit（1ファイル以上変更） | +1 |
| Pull Request（マージ済み） | +5 |
| Issue close（言語タグあり） | +2 |
| Star獲得（言語リポジトリ） | +0.1 |

MVPではCommitとPull Requestのみを採用する。Issue closeとStarは、ゲームバランスと不正対策を検討してから追加する。

## 領土・勢力の考え方

領土は、国取りゲームのような物理的な土地ではなく、技術分野ごとの主導権を表す。

例。

| 分野 | 強くなりやすいギルド例 |
|---|---|
| Web分野 | TypeScript |
| AI分野 | Python |
| System分野 | Rust / Zig / C++ |

GitHubでCommitやPull Requestを行うとCPが得られ、そのCPによって該当ギルドの勢力値が変化する。結果として、「どの技術がどの分野を支配しているか」が変わっていく。

MVPでは、領土は複雑なマップ操作ではなく、勢力図・ランキング・分野別の支配状況として可視化する。

## ユーザーフロー

```text
LP
↓
GitHub Login
↓
ユーザーデータ確認
├─ 既存ユーザー → Contribution解析
└─ 新規ユーザー → 初期プロフィール作成 → 推しギルド選択
↓
Contribution解析
↓
CP獲得
↓
ランキング・領土に反映
```

新規ユーザーは、最初に推しギルドを1つ選ぶ。既存ユーザーはログイン後、そのままContribution解析へ進む。

## 勝敗・サイクル

現時点では**シーズン制**を基本にする。永続蓄積だけだと後発ユーザーが参加しづらく、週次リセットだけだと積み上げ感が弱い。シーズン制なら、一定期間の熱量を競いつつ、過去の勝利を履歴として残せる。

シーズン終了時に、最も勢力値が高いギルドが勝利する。勝利したギルドは、次シーズンの世界設定、称号、見た目、イベントなどに反映される。

## キャラクター設定（案）

各ギルドは、キャラクター性を持つ。MVPでは見た目や能力値を作り込みすぎず、言語詳細画面に簡単なプロフィールとして表示する。

| ギルド | キャラ属性 | 強み |
|---|---|---|
| Rust | 鎧武者・無敵の守り | 安全性・速度 |
| Python | 魔法使い・万能型 | 汎用性・学習速度 |
| Go | Gopherギルド・シンプル特化 | 並列処理・起動速度 |
| Java | 重装騎士・堅牢 | 安定性・エコシステム |
| TypeScript | 変身戦士・適応型 | フロント〜バック |
| Haskell | 賢者・理論派 | 型安全性 |
| Zig | 原始の戦士・最低レイヤー | メモリ制御 |

## UXイメージ

ログインするとランキング画面が表示され、各ギルドの勢力順位が見える。自分のGitHubアカウントを連携すると、自分の活動がCPとして反映される。ギルドをクリックすると、そのギルドの勢力値、順位、簡単なキャラクター情報、最近のContributionログが見られる。

MVPの主要な画面は以下。

| 画面 | 役割 |
|---|---|
| ランディングページ | ゲームのコンセプトを伝え、GitHubログインへ誘導する |
| GitHubログイン | GitHub連携を開始する |
| 初期プロフィール | 新規ユーザーが推しギルドを選ぶ |
| ランキング | シーズン内のギルド勢力順位を見る |
| 自分の貢献 | GitHub活動が何CPになったかを見る |
| ギルド詳細 | ギルドの勢力値、順位、キャラクター情報、最近の活動を見る |

## MVP後の拡張候補

- CPを使ってギルド内に施設や記念像を建てる
- Gopher像のようなギルド固有の建築物を追加する
- ユーザーがCPを消費してギルドを発展させる
- 分野別マップをよりゲーム的な領土表現にする
- ギルドごとのイベントやバトルを追加する
- Issue close、Star、DiscussionなどもCP対象にする
- キャラクター能力値やスキルをランキングに反映する
